# JSON Pro 完整升级设计

- 日期：2026-03-30
- 主题：JSON Pro 工具站完整升级一轮（方案 B）
- 范围：在不引入后端的前提下，提升核心准确性、文件工作流和工程化基础

## 1. 目标

本轮升级聚焦三个方向：

1. 提升核心结果的正确性
   - 校验从“仅语法合法”升级为“语法 + Schema 结构校验”
   - TypeScript 类型生成从“示例推断”升级为“更接近真实数据结构的推断”
   - Diff 从“文本行差异”升级为“路径级语义差异”

2. 提升真实使用场景下的可用性
   - 增加文件导入、结果下载、剪贴板读取
   - 扩展格式转换能力，覆盖更常见的数据交换格式
   - 统一多个工具页的操作方式，降低学习成本

3. 补齐工程化基础
   - 增加 lint 与 type-check 脚本
   - 将 README 改造成真实项目说明，而不是模板默认文档

## 2. 非目标

本轮不做以下内容：

- 超大文件的 Web Worker 化
- 树视图虚拟滚动
- 协作分享链接
- 账户、云端同步、历史记录中心
- 后端服务或持久化存储

这些能力都可能有价值，但会明显扩大改造面，不适合作为本轮完整升级的范围。

## 3. 总体方案

本轮保持现有 Vue 3 + Vite 单页结构，不改变顶层导航与主要页面布局。

改造方式：

- 保留现有 Tab 结构，避免用户心智切换
- 新增一批可复用工具函数，收敛解析、序列化、下载、导入、语义比较逻辑
- 优先增强 `ValidateTab`、`TypesTab`、`DiffTab`、`ConvertTab`
- 对 `FormatTab` 等页面补充统一文件工作流能力
- 在工程层增加 ESLint 与 `vue-tsc` 的静态检查能力

## 4. 功能设计

### 4.1 Schema 校验

#### 目标
让校验页不仅告诉用户 JSON 能不能被解析，还能告诉用户“结构是否符合预期”。

#### 设计
在 `ValidateTab` 中保留现有 JSON 输入区，并新增：

- Schema 输入区
- 示例 Schema 按钮
- 校验模式说明
- 错误列表区域

校验分两层：

1. 语法校验
   - JSON 解析是否成功
2. Schema 校验
   - 当 Schema 输入非空时，对解析后的 JSON 再执行结构校验

#### 支持的 Schema 能力
本轮采用一个轻量可控的子集，而不是一次性覆盖完整 JSON Schema 标准：

- `type`
- `properties`
- `required`
- `items`
- `enum`
- `additionalProperties`
- `minLength` / `maxLength`
- `minimum` / `maximum`

#### 输出形式
- 顶部状态：合法 / 不合法
- 列表展示每个错误：
  - 路径
  - 规则
  - 错误说明
- 如果只有语法合法但 Schema 不通过，要明确区分“语法通过、结构不通过”

#### 取舍
不引入重量级完整 Schema 编辑器，先以文本输入 + 结构校验为主，避免 UI 和依赖过重。

### 4.2 TypeScript 类型生成增强

#### 目标
让生成结果在更多真实 JSON 场景下可直接使用，而不是只适合演示数据。

#### 设计
重写当前类型推断逻辑，新增以下规则：

1. 处理非法属性名
   - 非法标识符改为字符串字面量属性
   - 例如 `user-name` → `'user-name': string`

2. 处理数组
   - 空数组输出 `unknown[]`
   - 同类数组合并为单一元素类型
   - 异构数组输出联合类型数组，如 `(string | number | null)[]`

3. 处理对象数组
   - 合并多个对象元素的字段
   - 缺失字段推断为可选属性
   - 不同字段类型合并为联合类型

4. 处理 `null`
   - 与其他类型并存时形成联合类型，例如 `string | null`

5. 处理顶层数组与原始值
   - 允许输出 `type Root = ...`
   - 不强制所有场景都生成 `interface`

6. 去重与命名稳定性
   - 对嵌套对象生成稳定名称
   - 避免同名接口重复定义

#### 输出形式
- 优先生成接口 + 顶层类型别名组合
- 如果无法合理命名，则退回内联对象类型

### 4.3 语义级 Diff

#### 目标
让 Diff 结果体现“数据结构发生了什么变化”，而不是“格式化后的哪几行变化了”。

#### 设计
对两个 JSON 值做递归比较，生成路径级差异集合。

#### 差异类型
- `added`：新路径存在
- `removed`：原路径消失
- `changed`：同一路径值发生变化
- `type_changed`：同一路径类型变化

#### 路径格式
统一采用点路径 + 数组下标：

- `root.user.name`
- `root.items[0].price`

#### 比较规则
- 对象按键名比较，不受键顺序影响
- 数组默认按索引比较
- 原始值直接比较
- 对象与数组、数组与原始值之间视为类型变化

#### 结果展示
- 结果摘要：新增 / 删除 / 修改数量
- 主列表按差异项展示：
  - 类型标签
  - 路径
  - 旧值
  - 新值
- 提供复制文本摘要能力

#### 取舍
数组暂不引入“按 id 智能对齐”，因为这需要用户指定主键或增加复杂启发式，适合下一轮迭代。

### 4.4 格式转换扩展

#### 目标
将格式转换页从演示型工具提升为更常用的数据转换入口。

#### 新增支持格式
- JSON
- YAML
- XML
- CSV
- TOML
- URL Query

#### 统一中间模型
所有转换遵循：

1. 输入格式 → JS 值
2. JS 值 → 输出格式

这样可以避免为每一对格式单独实现转换矩阵。

#### 兼容边界
- CSV 仅支持对象数组、平铺对象数组、标量数组三类主场景
- TOML 优先支持对象、数组、标量，不追求覆盖全部 TOML 特性
- URL Query 主要面向扁平对象和浅层数组
- XML 保持当前对象映射方案，不解决复杂命名空间与混合内容问题

#### 输出形式
- 保留当前左右分栏
- 输入输出格式选择器扩展
- 错误信息明确指出“不支持的输入形态”而不是笼统失败

### 4.5 文件工作流

#### 目标
让页面不再只依赖手动粘贴。

#### 设计
新增一组可复用文件能力：

- 从本地文件导入文本
- 将结果下载为文件
- 从剪贴板读取文本

#### 首批接入页面
- `FormatTab`
- `ValidateTab`
- `ConvertTab`
- `TypesTab`
- `DiffTab`（左右侧都支持导入）

#### 行为约定
- 导入文件后直接写入当前输入框
- 下载文件时根据当前输出格式自动推断扩展名
- 剪贴板读取失败时给出明确错误提示

### 4.6 通用交互统一

#### 目标
减少不同页面之间的操作差异。

#### 设计
逐步统一以下操作：

- `示例`
- `清空`
- `导入`
- `读取剪贴板`
- `复制`
- `下载`

不强行一次性抽成大型公共组件，优先抽取通用工具函数与轻量复用片段，避免把当前简单结构改得过重。

## 5. 代码结构设计

### 5.1 新增工具模块
计划将通用逻辑拆到以下模块：

- `src/utils/schema.js`
  - 轻量 Schema 校验
- `src/utils/types.js`
  - TS 类型推断与输出
- `src/utils/diff.js`
  - 语义 Diff
- `src/utils/convert.js`
  - 多格式解析与序列化
- `src/utils/io.js`
  - 文件导入、下载、剪贴板读取

当前已有 `src/utils.js` 可保留通用基础函数，避免把所有逻辑继续堆在同一个文件。

### 5.2 页面改动

- `ValidateTab.vue`
  - 扩展为 JSON + Schema 双输入布局
- `TypesTab.vue`
  - 替换推断算法，补充导入/下载
- `DiffTab.vue`
  - 替换为语义对比结果视图
- `ConvertTab.vue`
  - 扩展格式与错误提示
- `FormatTab.vue`
  - 补充导入/下载/读取剪贴板
- `style.css`
  - 增加差异列表、工具栏与辅助状态样式
- `package.json`
  - 增加脚本与依赖
- `README.md`
  - 重写项目说明

## 6. 依赖策略

计划新增或调整的依赖：

- `toml`：TOML 解析与序列化
- `eslint`、`eslint-plugin-vue`、`@vue/eslint-config-prettier` 或等价基础组合：lint
- `typescript`、`vue-tsc`：type-check

CSV 与 Query 优先自行实现，避免为简单功能引入额外重量级依赖。

## 7. 错误处理

### 7.1 原则
- 明确区分“解析失败”和“转换目标不支持”
- 给出尽量接近用户输入位置或路径的反馈
- 避免只显示原始异常文本

### 7.2 具体策略
- Schema 校验错误显示路径与规则名
- 类型生成失败时尽量降级，而不是整个中断
- 格式转换对不支持的结构给出原因
- 文件导入失败时保留原输入，不覆盖用户内容

## 8. 测试与验证

本项目当前没有测试框架，本轮先建立最基础的静态校验底座：

- `npm run build`
- `npm run lint`
- `npm run type-check`

如果实现中新增的纯函数足够集中，优先写成可独立验证的工具函数结构，便于下一轮补单元测试。

## 9. 风险与应对

### 风险 1：轻量 Schema 校验覆盖度有限
应对：在 README 中明确支持子集，避免误导为完整 JSON Schema 实现。

### 风险 2：格式转换边界复杂
应对：先限定 CSV / TOML / Query 的支持场景，错误时输出明确说明。

### 风险 3：改造面较大
应对：优先把复杂逻辑下沉到工具模块，页面只保留交互与展示。

9. 重写 README
10. 运行构建与静态检查

## 11. 成功标准

完成后，项目应满足：

- 可对 JSON 执行语法 + Schema 双层校验
- TS 类型生成可正确处理异构数组、非法键名、可选字段与 `null`
- Diff 可输出路径级语义差异
- Convert 支持 JSON / YAML / XML / CSV / TOML / Query
- 至少核心页面支持导入、下载、读取剪贴板
- 项目具备 `build`、`lint`、`type-check` 基础脚本
- README 能真实说明项目用途与能力边界
